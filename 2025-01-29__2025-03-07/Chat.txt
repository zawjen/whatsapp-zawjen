For our use case of billions of requests, performance and testibility we have to consider a few things.


When comparing Express.js and NestJS for REST API development, considering performance, scalability, and testability is crucial. Express.js is a lightweight, unopinionated framework that offers maximum flexibility and is ideal for small to medium projects where simplicity and speed matter. However, as applications grow, managing scalability, structure, and maintainability becomes challenging. NestJS, built on top of Express (or Fastify for better performance), provides a modular and opinionated architecture, built-in dependency injection, TypeScript support, and native microservices compatibility, making it better suited for large-scale applications.

From a performance standpoint, Express has lower overhead and is slightly faster for simple APIs, but NestJS with Fastify can outperform it while retaining a well-structured framework. In terms of testability, NestJS has a significant advantageâ€”it comes with built-in support for unit testing, integration testing, and dependency injection, making it easier to mock dependencies and write isolated, maintainable tests using Jest or other testing frameworks. Express, being unopinionated, requires additional setup and third-party libraries for proper test coverage.

Overall, if you need a quick and flexible solution, Express is great, but for scalability, maintainability, built-in testability, and enterprise-grade APIs, NestJS is the better choice. <This message was edited>

2/16/25, 12:21â€¯PM - Rafey: We will not be using monorepos.

We have one repo for web, admin, app and each cli and microservice to offer independence and scalability.

Monorepos offer several advantages, particularly for teams managing multiple related projects. They enable code sharing and reusability, ensuring that common libraries and components are used across projects without duplication. Dependency management becomes more consistent, as all projects share the same package versions, reducing compatibility issues. Atomic changes allow updates to shared packages across all projects in a single commit, ensuring synchronization. Additionally, monorepos simplify collaboration by allowing developers to work on multiple projects without switching repositories and make CI/CD management more centralized and efficient. However, monorepos also come with challenges. Large repositories can cause performance issues, slowing down Git operations, dependency installations, and even IDEs. Managing dependencies in a monorepo can be complex, requiring tools like pnpm workspaces, Yarn workspaces, or Nx. As the repository grows, scalability issues arise, making permission control, CI/CD configuration, and access management more difficult. A significant risk is that a breaking change in a shared package can cause failures across multiple projects. Additionally, tracking history for individual projects becomes harder, and some tools may not fully support monorepo structures, requiring extra configuration. Overall, monorepos are ideal for closely related projects with shared dependencies and frequent collaboration, but they may not be the best choice for independent projects with different lifecycles or scalability concerns.

Pros: Next.js 15 offers improved performance with optimized server-side rendering (SSR) and static site generation (SSG). The new App Router enhances server components, making data fetching more efficient and reducing client-side JavaScript. Cons: The learning curve for the App Router and server components can be steep, especially for developers transitioning from older Next.js versions. Managing state in a server-heavy environment requires new patterns, which might not be intuitive at first.

React 19

Pros: React 19 brings better performance, improved concurrency features, and updates like Actions, which simplify form handling. It also enhances React Server Components (RSC), making full-stack development smoother.

Cons: Many libraries are still catching up with React 19's changes, meaning potential compatibility issues. Additionally, developers need to adapt to new patterns like Actions and RSC, which can be a shift from traditional React practices.

ShadCN UI

Pros: ShadCN UI provides beautifully designed, accessible components that are customizable without the usual bloat of UI frameworks like Material UI. Since itâ€™s built on Radix UI and Tailwind CSS, it ensures flexibility and responsiveness.

Cons: While highly customizable, it requires familiarity with Tailwind CSS, which may be a drawback for those used to traditional CSS or component libraries with predefined styles. Additionally, integrating it into a design system may need extra setup.

TanStack Query

Pros: TanStack Query simplifies API calls by handling caching, background fetching, and automatic updates. It significantly reduces boilerplate compared to traditional state management solutions for fetching and storing remote data.

Cons: The mental shift from Redux or simple useState/useEffect patterns can be challenging. Overuse may lead to unnecessary complexity in cases where simple state management would suffice.

Motion React

Pros: Motion React (Framer Motion) is an elegant, easy-to-use animation library with great performance optimizations. It integrates well with React and allows for smooth, declarative animations without requiring deep knowledge of CSS animations.

Cons: While lightweight, it still adds some overhead to bundle size. Overusing animations can also affect performance, especially if not optimized properly.

React Hook Form

Pros: React Hook Form optimizes form management by minimizing re-renders and providing built-in validation. Itâ€™s lightweight and integrates well with other libraries, including UI frameworks and state management tools.

Cons: The API, while powerful, can be unintuitive for beginners. Complex forms may require additional learning to fully leverage features like field arrays and custom validations.

Overall Verdict

This stack is incredibly powerful for building modern web apps. However, it requires developers to stay updated with newer paradigms, especially with Next.js 15 and React 19 evolving rapidly. If used wisely, it provides a scalable, performant, and developer-friendly experience.

Prisma ORM and Its Migrate Process

Prisma is a modern ORM for Node.js and TypeScript that provides a type-safe, database-first approach to working with relational databases like PostgreSQL. It simplifies database interactions through its intuitive Prisma Client, while Prisma Migrate handles schema migrations. Prisma Migrate allows developers to define changes in a schema.prisma file and generate migration scripts using prisma migrate dev for development or prisma migrate deploy for production. It automatically creates versioned migration files, ensuring smooth schema upgrades and rollbacks. To downgrade, developers can use prisma migrate resolve to roll back to a previous migration, preserving data integrity. For seeding data, Prisma provides a prisma db seed command that runs a custom script, allowing developers to populate databases with test or initial data easily.

Elasticsearch ORM and Its Upgrade, Downgrade, and Seed Process

Unlike relational databases, Elasticsearch does not have a traditional ORM, but developers interact with it using the official Elasticsearch Node.js client (@elastic/elasticsearch) or libraries like Elasticsearch ORM (elastic-orm), which offer a structured way to query data. Since Elasticsearch is schema-less, upgrading an index involves creating a new versioned index (e.g., products_v2), using the Reindex API to copy data from the old index, and switching an alias to the new index. Downgrading follows the same process, reindexing data back to the previous version. Seeding data is done via the Bulk API, which allows inserting multiple documents in a single request, making it efficient for initial data population. Since indices cannot be modified like SQL tables, schema changes require index recreation, making versioning essential for Elasticsearch migrations.

3/1/25, 1:02â€¯PM - Rafey: A very good UX suggestion from @923352807751 about Storybook 

I used a Storybook in one of the financial product AARP Money Map. Because it was just one website without much reuse, developers often forget to update Storybook and it stays in a crashed state. So If not maintained properly, the Storybook version of components might differ from real app behavior.

The good thing about Storybook is that people across the world can Develop and test UI components in isolation without worrying about app dependencies. It has Powerful plugins like accessibility testing, viewport adjustments, and interaction testing. The best thing perhaps is that Designers and product managers can review components without needing to run the full app.

So skip Storybook if you're working on a simple project, a backend-heavy app, or if maintaining separate stories is too much overhead.

Use Storybook if you're working on a large-scale app like Zawjen, design system, or reusable UI library. It improves modularity and speeds up UI development. Zawjen is not a design system, or reusable UI library but with our efforts we can give world the best 
design system, or reusable UI library.

People around the world are seeing your open source work and your next great opportunity lies in building a great design system, or reusable UI library. 

Zawjen will have dozens and dozens of websites and apps eventually for internal and external use.

I recommend Storybook.
